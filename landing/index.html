AI Gateway

A Zero Trust reference design for SBIR/CMMC-bound teams who need defensible AI tooling without enterprise vendor pricing — one authenticated API gateway to every frontier model and your government resources.

What this is

This is a reference design, not a product — a defensible Zero Trust AI architecture, assembled from Cloudflare Zero Trust, AWS, Okta, and open-source components. At its center is one authenticated AI API gateway: a user signs in once and gets a single scoped key that reaches every frontier model (OpenAI + Anthropic) and their government / compliance resources — SAM.gov, NIST 800-53, CMMC, POA&Ms — through one OpenAI-compatible endpoint, governed centrally.

It demonstrates the controls a defense-industrial-base or SBIR-stage team actually needs around that gateway: identity-aware access with MFA, no public ingress, inline AI guardrails, workload egress allowlisting, and an MCP server that queries your own compliance evidence.

It answers the question "what does a defensible Zero Trust AI architecture look like at commodity prices?" — roughly $95/month at idle on AWS. Full Terraform, 13 ADRs, structured audit evidence at every control point.

Architecture

The gateway, concretely

The gateway has two demonstrable surfaces — an API and a control plane. Neither is a chat window.

  • An OpenAI-compatible endpoint. The only change from calling OpenAI directly is the base_url. One scoped virtual key reaches every frontier model (gpt-4o, claude-opus-4-8, claude-sonnet-4-6, claude-haiku-4-5, claude-fable-5) and the government-resource tools (a live sam_gov_lookup, NIST 800-53, CMMC, POA&Ms). An injection prompt is blocked pre-call by the NeMo guardrail — no provider request, no spend.
  • The LiteLLM Admin control plane (SSO-gated, at /ui). This is the screen that is the gateway: virtual keys, per-team budgets and live spend, request logs, guardrail blocks, and model + tool routing.

Open WebUI (chat.optimallabs.io) is one labeled example of a client app consuming this gateway — not the gateway, and not the product.

What's worth defending here

  • Two-pattern SSO split. Open WebUI uses trusted-header SSO (the edge proves identity); the LiteLLM admin panel runs its own OIDC handshake. Match the auth mechanism to the blast radius.
  • Read-only MCP first. The compliance MCP exposes only read tools. A prompt-injection success can read seeded lab data — not mutate your evidence store. Write mode is a separate, gated project.
  • Detection-as-a-service guardrails. Inline secret, PII (Luhn-checked), and prompt-injection detection on every request — fail-closed.
  • Workload egress allowlisting. The part most cloud teams skip: app hosts can't reach the internet except through a default-deny domain allowlist. No SSH; SSM-only admin; no public ingress to any instance.
  • Compliance teams should run MCP against their own evidence — POA&Ms, the NIST catalog, CMMC status, and a real SAM.gov lookup — inside a gateway they control, not a vendor's API.
